# 1. SQL 성능 튜닝 절차

``` text
느린 SQL 발견
      ↓
실행 계획(EXPLAIN)
      ↓
인덱스 확인
      ↓
JOIN 방식 분석
      ↓
SQL 개선
      ↓
재검증
```

원칙 - 필요한 컬럼만 조회 - 불필요한 Full Scan 최소화 - 인덱스 활용 -
통계정보 최신화

# 2. Index 심화

  종류            특징                사용 사례
  --------------- ------------------- ---------------
  Clustered       데이터 자체 정렬    PK
  Non-Clustered   별도 인덱스         일반 조회
  Composite       여러 컬럼           복합 조건
  Covering        인덱스만으로 조회   SELECT 최적화

주의사항

-   인덱스가 많을수록 INSERT/UPDATE 성능 저하
-   선택도(Cardinality)가 높은 컬럼에 효과적

------------------------------------------------------------------------

# 3. Execution Plan

확인 항목

-   Full Table Scan
-   Index Scan
-   Join Method
-   Cost
-   Estimated Rows

대표 명령

``` sql
EXPLAIN SELECT * FROM employee;
```

------------------------------------------------------------------------

# 4. MVCC

MVCC(Multi-Version Concurrency Control)는 여러 버전의 데이터를 유지하여
읽기와 쓰기의 충돌을 줄인다.

장점

-   높은 동시성
-   읽기 성능 향상
-   Lock 경쟁 감소

대표 DBMS

-   PostgreSQL
-   MySQL(InnoDB)

------------------------------------------------------------------------

# 5. Lock

  종류             설명
  ---------------- -------------
  Shared Lock      읽기
  Exclusive Lock   쓰기
  Row Lock         행 단위
  Table Lock       테이블 단위

------------------------------------------------------------------------

# 6. Deadlock

Deadlock은 둘 이상의 트랜잭션이 서로의 Lock 해제를 기다리는 상태이다.

예방 방법

-   동일한 순서로 테이블 접근
-   트랜잭션 최소화
-   Lock 시간 단축
-   인덱스 최적화

------------------------------------------------------------------------

# TOPCIT 실전 문제

## 문제1 (중 \| ★★★★★)

SQL 튜닝 시 가장 먼저 확인해야 할 것은?

① README ② 실행 계획(EXPLAIN) ③ DB 재설치 ④ Trigger 삭제

**정답:** ②

**해설:** 실행 계획은 Full Scan, Join 방식, 인덱스 사용 여부를 확인하는
출발점이다.

------------------------------------------------------------------------

## 문제2 (상 \| ★★★★★)

MVCC의 가장 큰 장점은?

① 디스크 용량 감소 ② 동시성 향상 ③ 네트워크 속도 증가 ④ CPU 오버클럭

**정답:** ②

------------------------------------------------------------------------

## 문제3 (상 \| ★★★★★)

두 개 이상의 컬럼을 함께 검색하는 조건에 적합한 인덱스는?

① Clustered Index ② Composite Index ③ Bitmap Index ④ Unique Key

**정답:** ②

**해설:** Composite Index는 다중 컬럼 조건 검색에 적합하다.

------------------------------------------------------------------------

## 문제4 (시나리오 \| ★★★★★)

트랜잭션 두 개가 서로 Lock을 기다리며 종료되지 않는다. 이 현상은?

① Starvation ② Deadlock ③ Fragmentation ④ Replication

**정답:** ②

------------------------------------------------------------------------

## 문제5 (시나리오 \| ★★★★★)

SELECT 성능은 빠르지만 INSERT가 매우 느려졌다. 가장 먼저 의심할 것은?

① 인덱스 과다 생성 ② DNS 오류 ③ DHCP 장애 ④ RAID0

**정답:** ①

**해설:** 과도한 인덱스는 데이터 변경 시 유지 비용을 증가시킨다.
